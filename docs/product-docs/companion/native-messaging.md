# Native Messaging 与扩展配对

## 本页说明

本页说明扩展如何优先识别本机 Companion，以及在什么情况下，你才需要把这条配对路径当作排查重点。

## 先怎么理解这项能力

对普通用户而言，Native Messaging 不是一个需要单独学习或配置的功能，而是扩展识别本机 Companion 的标准自动配对路径。

它的作用很直接：

- 让扩展优先自动发现本机 Companion
- 尽量减少手动填写连接信息
- 让安装完成后的接入更接近自动识别

也就是说，这项能力的目标不是增加一层设置，而是减少普通用户的连接负担。

## 正常使用时，你通常不需要做什么

如果 Companion 已经安装完成，且扩展能够正常识别它，那么你通常不需要：

- 单独配置 Native Messaging
- 手动理解底层配对机制
- 在第一次使用时进入这页排查

对普通用户来说，只要扩展已经识别 Companion，这条配对路径就可以视为正常工作。

## 什么时候才需要关心这页

通常只有在下面这些情况里，你才需要把这页当成重点：

- 安装已经完成，但扩展里仍然看不到 Companion
- 自动配对持续失败
- 诊断或自检提示浏览器识别、配对或本机注册存在问题
- 你已经确认安装来源无误，也处理过系统安全提示，但接入仍未完成

如果你当前遇到的是安装包被系统拦截，例如 macOS Gatekeeper 或 Windows SmartScreen 阻止安装，那么这还不是配对问题，应先回到[Companion 连接问题](../troubleshooting/companion-connection.md)处理安装放行。

## 这条配对路径和诊断页是什么关系

对普通用户来说，可以这样理解：

- 安装页负责说明标准安装顺序
- 这页负责解释扩展为什么通常可以自动识别本机 Companion
- 诊断页负责在自动识别失败后，继续判断问题属于哪一类

其中最关键的一点是：自动配对失败，不等于安装失败。只有在安装已经完成、但扩展仍未识别 Companion 时，这条配对路径才真正成为排查重点。

## 推荐处理顺序

如果你怀疑问题出在自动配对，建议按下面顺序处理：

1. 先确认问题不是安装放行问题。
2. 再回到扩展中的 Companion 页面重新检测。
3. 如果仍未识别，再进入[诊断与自检](diagnostics-and-self-check.md)，查看浏览器识别或配对相关结果。
4. 只有在方向已经明确时，再进入修复或手动连接流程。

这样做的目的是先分清问题类别，再处理配对，而不是一开始就把所有异常都归到 Native Messaging 上。

## 如果自动配对失败，不要先做什么

更稳妥的做法是避免下面这些常见误判：

- 不要把自动配对失败直接等同于安装包异常
- 不要在安装已完成的情况下，第一步就重复下载安装
- 不要在还没有判断问题类别时，直接做大范围修复

先分清是安装放行、自动识别还是后续连接问题，排查效率会更高。

Ghast AI 通过 Native Messaging 作为扩展优先识别本机 Companion 的标准自动配对路径。对普通用户而言，这不是一个需要单独配置的功能，而是安装完成后默认工作的识别机制；只有在安装已经完成但扩展仍无法识别 Companion 时，才需要把它作为重点排查对象。
