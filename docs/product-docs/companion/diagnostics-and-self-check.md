# 诊断与自检

## 本页说明

本页说明在 Companion 已经安装完成之后，如果出现识别异常、连接异常、工作区不可用或本地能力不稳定，应该如何使用健康检查、自检和修复功能。

## 什么时候应该先看这页

当你遇到下面这些情况时，应优先进入这页：

- 安装已经完成，但扩展里看不到 Companion
- 自动配对没有成功
- 工作区之前可以使用，现在不可用
- 本地能力表现不稳定，或结果明显不完整
- 技能、本地工具或高级能力提示本机环境不完整

如果你当前遇到的是“安装包被系统拦截”，例如 macOS Gatekeeper 或 Windows SmartScreen 阻止安装，那么这还不是诊断阶段，应先回到[Companion 连接问题](../troubleshooting/companion-connection.md)处理安装放行问题。

## 这页里的三类功能

| 功能 | 应该怎么理解 |
| --- | --- |
| 健康检查 | 先确认 Companion 当前是不是在线 |
| 自检 | 帮你判断问题更接近配置、配对、工作区还是本机环境 |
| 修复 | 在问题已经基本定位后，针对常见问题执行恢复动作 |

对普通用户来说，不需要先理解底层机制，只需要按顺序使用这三类功能。

## 自检主要会看什么

自检当前会重点检查这些内容：

- 本地配置是否可读
- Companion 访问凭据是否存在
- 当前工作区边界是否有效
- 浏览器识别 Companion 所需的本机注册是否完整
- 已配置的 MCP 或本地工具是否具备可执行条件

你可以把这一步理解为：不是马上修，而是先确认“问题大概落在哪一层”。

## 推荐的处理顺序

最稳妥的顺序如下：

1. 先做健康检查，确认 Companion 是否在线。
2. 再做自检，判断问题更接近连接、工作区、配对还是环境缺失。
3. 只有在方向已经明确时，再执行修复动作。

这样做的目的，是先确认状态，再缩小范围，最后才做修复，避免把安装、连接和环境问题混在一起处理。

## 修复功能什么时候用

修复功能更适合在你已经确认“当前确实存在问题”时使用，而不是每次出现异常就直接执行。

更合适的理解方式是：

- 健康检查负责确认状态
- 自检负责帮助定位
- 修复负责处理已经确认的问题

当前常见修复通常围绕两类问题展开：

- 补齐或恢复本地配置
- 重新建立浏览器对 Companion 的识别与配对

如果你只是还没有判断清楚问题属于哪一类，不建议先做修复。

## 仍然无法恢复时

如果你已经完成健康检查、自检和必要修复，但问题仍然存在，建议按问题类型继续分流：

- 安装或系统放行问题：查看[Companion 连接问题](../troubleshooting/companion-connection.md)
- 标准安装顺序与自动识别：查看[安装与自动配对](install-and-auto-pair.md)
- 工作区边界或访问范围问题：查看[工作区权限问题](../troubleshooting/workspace-permissions.md)

如果你已经进入高级排查阶段，也可以使用 `trapezohe-companion self-check` 查看当前自检结果，并按建议执行修复动作。

Ghast AI 通过 Companion 的健康检查、自检和修复路径，为本地能力提供统一的用户排查入口。对普通用户而言，推荐顺序始终是先确认状态、再判断问题类别、最后执行修复；如果安装包本身还未被系统放行，则应先处理安装放行问题，而不是提前进入诊断流程。
