# Docs Pages Deployment

## Overview

This document defines the staging and production deployment path for the documentation subsite at `doc.trapezohe.ai`.

The repository is prepared for a Cloudflare Pages **Direct Upload** workflow driven by GitHub Actions. This keeps the staging and production rules in version control instead of relying only on dashboard settings.

## Deployment model

| Environment | Source | Cloudflare Pages branch | URL |
| --- | --- | --- | --- |
| Staging | Push to `docs-staging` or manual workflow dispatch with `staging` | `staging` | Preview domain generated by Pages, typically `https://staging.ghast-docs.pages.dev` |
| Production | Push to `main` or manual workflow dispatch with `production` | `main` | `https://doc.trapezohe.ai` |

The preview-domain pattern above is an inference from Cloudflare Pages branch preview behavior and assumes the project name remains `ghast-docs`.

## Repository configuration

The repository now includes:

- GitHub Actions workflow: `.github/workflows/docs-pages.yml`
- Docs verification script: `scripts/check-doc-links.py`
- Verification commands in `package.json`
- Static headers for the docs site in `docs/product-docs/public/_headers`

Use these local commands before deployment:

```bash
pnpm install
pnpm docs:build
pnpm docs:check-links
pnpm docs:verify
```

## Cloudflare Pages project settings

Use these values in Cloudflare Pages:

```text
Project name: ghast-docs
Production branch: main
Framework preset: None
```

This setup assumes Direct Upload deployment through GitHub Actions rather than Cloudflare Git integration.

## GitHub secrets

Add these repository secrets before enabling the workflow:

```text
CLOUDFLARE_API_TOKEN
CLOUDFLARE_ACCOUNT_ID
```

For the current setup, the token should be a custom Cloudflare API token scoped to:

```text
Account / Cloudflare Pages / Edit
```

## Custom domain

After the first successful production deployment:

1. Open the `ghast-docs` Pages project.
2. Add the custom domain `doc.trapezohe.ai`.
3. Complete the DNS validation in Cloudflare.

Keep staging on the Pages preview domain unless you later decide to add a separate staging hostname.

## Release flow

Recommended release sequence:

1. Push the docs changes to `docs-staging`.
2. Confirm the preview deployment is healthy on the Pages preview URL.
3. Verify `/cn/`, `/en/`, and root redirect behavior.
4. Merge or cherry-pick the same change set to `main`.
5. Let the workflow publish the production deployment to `doc.trapezohe.ai`.

## Expected runtime behavior

- Root path `/` redirects to `/cn/`
- Chinese docs resolve under `/cn/`
- English docs resolve under `/en/`
- Shared assets are served from the built VitePress output
- `_headers` applies static security and caching headers at the edge
